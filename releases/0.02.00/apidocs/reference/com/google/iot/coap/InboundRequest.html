public interface

InboundRequest

com.google.iot.coap.InboundRequest

Class Overview

Object representing an inbound request, providing methods for examining the request and responding to it. InboundRequests are handled by InboundRequestHandlers.

These objects are only valid within the initial call stack from which they are provided, UNLESS responsePending() has been called. In other words, it is a runtime error to make a call to a method of an instance of this class from outside of the initial call stack (such as from a separate thread) unless responsePending() was called from that initial call stack first. Effectively this means that if you want to defer responding to a request until a later point in time, you MUST call responsePending() before returning.

The InboundRequest interface also has a mechanism for reading Options in a way that keeps track of the index in the option list, much like an iterator. This mechanism is used to read the request's options (like URI_PATH) and process them as necessary.

This mechanism is also used to indicate to the InboundRequestHandler at what position in the path that it should consider itself hosted at. Take the following example using Resource instances: 


     // Set up our root ("/") resource
     Resource rootResource = new Resource();
     server.setRequestHandler(rootResource);

     // Add the folder "/test/"
     Resource<InboundRequestHandler> testFolder = new Resource<>();
     rootResource.addChild("test", testFolder);

     // Add a handler for "/test/hello"
     InboundRequestHandler helloHandler = new InboundRequestHandler() {
         public void onInboundRequest(InboundRequest inboundRequest) {
             inboundRequest.sendSimpleResponse(Code.RESPONSE_CONTENT, "Hello, World!");
         }
     }
     testFolder.addChild("hello", helloHandler);

When an inbound request for "/test/hello" arrives, rootResource looks at the first URI_PATH and uses that to look up the child: testFolder in this case. It then calls testFolder.onInboundRequest(...). At that point, testFolder needs to look up the next URI_PATH, but how does it know what the next URI_PATH would be? testFolder has no information about where it is rooted in the resource tree— it just gets a called to handle an inbound request, which could be from anywhere.

nextOption() provides the necessary context to make this arrangement work. rootResource "consumes" all of the options up to and including the option that describes the name the child in question— before calling testFolder.onInboundRequest(...). testResource uses that starting point as its own "root" for looking up its own children.

public abstract void acknowledge ()

Acknowledges the request. Sends an empty acknowledgement packet to the source of the request, indicating to the source that they do not need to retransmit the request but should keep waiting for a full response.

Throws
InboundRequestOutOfScopeException if this method was called from an unexpected thread without having called responsePending() from the original thread

public abstract boolean didAcknowledge ()

Indicates if this request has been acknowledged (piggy-backed or empty).

Most responses are sent piggy-backed with an ACK packet, but an empty ACK packet may be emitted by calling acknowledge() (or it may be spontaneously emitted after a call to responsePending()).

This method will return true in either case.

Returns
  • true if any sort of response has been sent, false otherwise.
